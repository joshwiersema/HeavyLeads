  <files>tests/scraper/pipeline.test.ts</files>
  <action>
The 6 failing tests in `tests/scraper/pipeline.test.ts` fail because the mock `returning()` always yields `{ id: "mock-lead-id-001" }`. When the pipeline flatMaps `results.newLeadIds`, all entries are the same ID. After dedup (Set or similar), `allNewLeadIds.length` is 0 or 1, so `deduplicateNewLeads` is never called.

Fix: Change the mock's `returning()` implementation to yield unique IDs on each call. Two approaches (use whichever fits the existing mock structure best):

**Option A (preferred):** Use a counter or `crypto.randomUUID()`:
```typescript
returning: vi.fn().mockImplementation(() =>
  Promise.resolve([{ id: `mock-lead-id-${crypto.randomUUID()}` }])
),
```

**Option B:** Use `mockResolvedValueOnce` for each expected call:
```typescript
returning: vi.fn()
  .mockResolvedValueOnce([{ id: "mock-lead-id-001" }])
  .mockResolvedValueOnce([{ id: "mock-lead-id-002" }])
  .mockResolvedValue([{ id: "mock-lead-id-003" }]),
```

The key requirement: each call to `returning()` must produce a distinct `id` so that `allNewLeadIds` after flatMap has the correct count and `deduplicateNewLeads` gets called.

Also ensure `beforeEach(() => { vi.clearAllMocks(); })` is present in the describe block to prevent mock state leaking between tests.

CONSTRAINT: Only modify the test file. Do NOT touch `src/lib/scraper/pipeline.ts` or any production code.
