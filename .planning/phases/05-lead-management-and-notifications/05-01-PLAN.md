    1. Create `src/actions/lead-status.ts` ("use server"):
       - `updateLeadStatus(leadId: string, status: LeadStatus)`: Get session (auth guard pattern from settings.ts), validate status is one of the 5 valid values using Zod, upsert into leadStatuses using `onConflictDoUpdate` targeting the unique index columns [userId, leadId, organizationId], set status and updatedAt. Call `revalidatePath("/dashboard")` and `revalidatePath(\`/dashboard/leads/${leadId}\`)`.
       - `getLeadStatus(leadId: string)`: Get session, query leadStatuses for the userId+leadId+orgId, return status string or "new" if no row.

    2. Create `src/actions/bookmarks.ts` ("use server"):
       - `toggleBookmark(leadId: string)`: Get session, check if bookmark exists for userId+leadId+orgId. If exists, delete it (return { bookmarked: false }). If not, insert with ON CONFLICT DO NOTHING (return { bookmarked: true }). Call `revalidatePath("/dashboard")`.
       - `getBookmarkedLeads()`: Get session, query bookmarks joined with leads for userId+orgId, return lead IDs array.

    3. Create `src/actions/saved-searches.ts` ("use server"):
       - `createSavedSearch(data: SavedSearchInput)`: Get session, validate with Zod schema (name required, all filter fields optional), insert into savedSearches, revalidatePath.
       - `deleteSavedSearch(id: string)`: Get session, delete where id AND userId AND orgId match (security: user can only delete their own), revalidatePath.
       - `getSavedSearches()`: Get session, select all for userId+orgId ordered by createdAt desc.
       - `getSavedSearchById(id: string)`: Get session, select by id AND userId AND orgId.
       - `savedSearchToParams(search: SavedSearchRow)`: Pure function (not a server action) that converts a saved search row into URLSearchParams string. Maps equipmentFilter to comma-separated "equipment" param, radiusMiles to "radius", keyword to "keyword", dates to ISO strings for "dateFrom"/"dateTo", sizes to "minProjectSize"/"maxProjectSize". Skips null/undefined values.

    IMPORTANT: Every query against lead_statuses, bookmarks, and saved_searches MUST include both userId AND organizationId in WHERE clauses -- missing organizationId causes cross-tenant data leaks.
