**types.ts**: Define types consumed by all scoring modules AND the query layer.
```typescript
import type { Industry } from "@/lib/onboarding/types";

export interface OrgScoringContext {
  industry: Industry;
  hqLat: number;
  hqLng: number;
  serviceRadiusMiles: number;
  specializations: string[];
  preferredLeadTypes: string[];
  targetProjectValueMin: number | null;
  targetProjectValueMax: number | null;
}

export interface LeadScoringInput {
  lat: number | null;
  lng: number | null;
  projectType: string | null;
  sourceType: string;
  applicableIndustries: string[];
  estimatedValue: number | null;
  valueTier: string | null;
  severity: string | null;
  deadline: Date | null;
  scrapedAt: Date;
}

export interface ScoreDimension {
  name: string;
  score: number;
  maxScore: number;
  reasons: string[];
}

export interface ScoringResult {
  total: number;
  dimensions: ScoreDimension[];
  matchReasons: string[];
}
```

**distance.ts**: Pure function `scoreDistance(distanceMiles: number | null, serviceRadiusMiles: number): ScoreDimension`. Uses the haversineDistance function from `@/lib/leads/queries` (already exported). The distance value is pre-computed before calling this function. Scoring tiers:
- null distance (no coords) = 0pts, reason "Location unknown"
- <= 10mi = 25pts, reason "{N} miles from your HQ"
- <= 25mi = 20pts
- <= 50mi = 15pts
- <= serviceRadius = 10pts
- <= serviceRadius * 1.5 = 5pts, reason "Just outside your {R}-mile service area"
- > 1.5x = 0pts, reason "Outside your service area"

**relevance.ts**: Pure function `scoreRelevance(lead: LeadScoringInput, org: OrgScoringContext): ScoreDimension`. Logic:
- +15 if lead.projectType matches any org.specializations (case-insensitive partial match), reason "Matches your {spec} specialization"
- +10 if org.industry is in lead.applicableIndustries, reason "Relevant to {industry} industry"
- +5 if lead.applicableIndustries has entries but org.industry is not in it (cross-industry), reason "Cross-industry opportunity"
- +5 if lead.sourceType is in org.preferredLeadTypes (map sourceType to lead type names using a lookup), reason "From your preferred {type} sources"
- -10 if lead.applicableIndustries has entries and org.industry not in it and no specialization match, reason "Outside your specializations"
- Clamp to 0-30.

**value.ts**: Pure function `scoreValue(lead: LeadScoringInput, org: OrgScoringContext): ScoreDimension`. Logic:
- estimatedValue is null and valueTier is null = 10pts "Value unknown"
- Within target range (min..max) = 20pts "Within your ${min}-${max} target range"
- Above target max = 15pts "Above your target range"
- Below 50% of target min = 5pts "Below your target range"
- Well below (below 25% of min) = 0pts "Well below target range"
- No target range configured = 10pts for any value

**freshness.ts**: Pure function `scoreFreshness(scrapedAt: Date): ScoreDimension`. Uses current time. Tiers:
- Today (<24h) = 15pts "Discovered today"
- <3 days = 12pts "Discovered {N} days ago"
- <7 days = 9pts
- <14 days = 6pts
- <30 days = 3pts
- Older = 0pts "Over 30 days old"

**urgency.ts**: Pure function `scoreUrgency(lead: LeadScoringInput): ScoreDimension`. Logic checks multiple urgency signals and takes the highest:
- sourceType === "storm" and deadline within 48h = 10pts "Active storm alert"
- sourceType === "bid" and deadline within 14d = 10pts "Bid deadline in {N} days"
- severity === "violation" = 8pts "Code violation -- mandatory fix"
- sourceType === "permit" = 5pts "Active building permit"
- sourceType === "incentive" and deadline within 30d = 10pts "Incentive expiring in {N} days"
- Default = 0pts

**engine.ts**: Orchestrator function `scoreLeadForOrg(lead: LeadScoringInput, org: OrgScoringContext, distanceMiles: number | null): ScoringResult`. Calls all 5 dimension functions, sums scores, clamps 0-100, collects all non-empty reasons into `matchReasons`. Export this as the main scoring entry point.

**index.ts**: Re-export from engine.ts and types.ts.

Write tests FIRST in `tests/scoring/engine.test.ts` covering:

3. Relevance awards +15 for specialization match, -10 for outside
4. Value awards 20 for in-range, 10 for unknown
5. Freshness decays correctly (today=15, 3d=12, etc.)
6. Urgency picks highest signal (storm trumps permit)
7. Match reasons are human-readable strings (not empty)
8. Total is clamped 0-100

Use `vi.useFakeTimers()` to control date-dependent tests (freshness).

**Update `src/lib/leads/queries.ts`:**

Add a new `getFilteredLeadsCursor` function. Do NOT remove existing functions (getFilteredLeads, getFilteredLeadsWithCount, getLeadById, etc.) -- they are consumed by bookmarks, digests, and other pages.

New interface:
```typescript
export interface CursorPaginationParams {
  cursor?: string; // lead ID -- fetch leads after this ID
  limit?: number;  // default 20
}

export interface GetLeadFeedParams extends CursorPaginationParams {
  orgId: string;
  userId: string;
  // Filters
  sourceTypes?: string[];
  maxDistanceMiles?: number;
  minValue?: number;
  maxValue?: number;
  projectTypes?: string[];
  dateFrom?: Date;
  dateTo?: Date;
  matchingSpecializationsOnly?: boolean;
  sortBy?: "score" | "distance" | "value" | "date";
  keyword?: string;
}

export interface LeadFeedResult {
  leads: ScoredLead[];
  nextCursor: string | null;
  hasMore: boolean;
}
```

Implementation of `getFilteredLeadsCursor`:
1. Query the organization table to get `industry`. Query organization_profiles to get full org scoring context (hqLat, hqLng, serviceRadiusMiles, specializations, preferredLeadTypes from profile data, targetProjectValueMin/Max).
2. Build an `OrgScoringContext` from this data.
3. Build SQL WHERE conditions:
   - `isNotNull(leads.lat)` and `isNotNull(leads.lng)` (need coords for distance)
   - Haversine distance within `maxDistanceMiles ?? org.serviceRadiusMiles * 1.5` (fetch slightly beyond radius so borderline leads can be scored)
   - If `cursor` provided: `leads.id > cursor` (for stable cursor ordering by ID)
   - If `sourceTypes` provided: `inArray(leads.sourceType, sourceTypes)`
   - If `minValue`/`maxValue`: range filter on `leads.estimatedValue`
   - If `projectTypes`: `inArray(leads.projectType, projectTypes)`
   - If `dateFrom`/`dateTo`: range on `leads.scrapedAt`
   - If `matchingSpecializationsOnly`: filter where `leads.applicableIndustries` overlaps with org.industry (use SQL `&& ARRAY[industry]::text[]` operator via `sql` tag)
   - If `keyword`: ilike across title, description, address
4. Fetch `BATCH_SIZE` (50) leads ordered by `leads.id ASC` (stable cursor ordering).
5. For each fetched lead, compute `distanceMiles` using the existing `haversineDistance` function, then call `scoreLeadForOrg(leadInput, orgContext, distanceMiles)`.
6. Sort the scored batch by the requested `sortBy` (default: score DESC, tiebreak scrapedAt DESC).
7. Take the first `limit` (default 20) leads.
8. Set `nextCursor` to the last returned lead's ID if there are more leads (batch was full).
9. Return `{ leads: scoredLeads, nextCursor, hasMore }`.

Also add `getLeadByIdScored(id: string, orgId: string): Promise<ScoredLead | null>` that fetches a single lead and scores it for the org. This will be used by the lead detail page in Plan 03.

IMPORTANT: Keep the existing `haversineDistance`, `getFilteredLeads`, `getFilteredLeadsWithCount`, `getLeadById`, `getLeadsByIds`, `getLeadSources`, and all helper functions. Only ADD new functions. The old pagination and scoring code is still used by bookmarks page, digest emails, etc. -- migrating those is out of scope for this phase.
