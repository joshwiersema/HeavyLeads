<objective>
Enhance cross-source deduplication to catch the same permit appearing from overlapping city and county Socrata/ArcGIS portals.

Purpose: The current `deduplicateNewLeads()` already handles geographic+text fuzzy matching, but it only compares new leads against ALL existing leads. The issue described in PERF-04 is that city and county portals produce leads with different `sourceId` values but identical real-world projects. The existing dedup logic DOES handle this (it queries candidates by bounding box on lat/lng, not by sourceId), but the candidate query uses a tight 0.002-degree bounding box (~0.14 miles) and only checks address OR title similarity. This plan widens the matching to also consider permit date proximity and permit number similarity for cross-portal dedup.

Output: Enhanced dedup logic that catches cross-source duplicates via permit-number fuzzy match and date+address compound matching.
</objective>

**2. Add a permit number normalization function** (exported for testing):

```typescript
/**
 * Normalize permit numbers for cross-source comparison.
 * Strips common prefixes like "BP-", "BLD-", "BLDG-", "COM-", "RES-",
 * removes dashes and spaces, lowercases.
 * E.g., "BP-2024-12345" and "2024-12345" and "202412345" all normalize
 * to "202412345".
 */
export function normalizePermitNumber(pn: string | null): string {
  if (!pn) return "";
  return pn
    .toLowerCase()
    .replace(/^(bp|bld|bldg|com|res|pmt|permit)[- ]*/i, "")
    .replace(/[-\s]/g, "")
    .trim();
}
```

**3. Enhance `isLikelyDuplicate`** to add a third matching path beyond address or title similarity. The function currently requires geographic proximity AND (address similarity OR title similarity). Add:

- **Permit number match**: If both candidates have non-empty normalized permit numbers and they match with similarity > 0.8, that alone (plus geographic proximity) is sufficient.
- **Date + address compound match**: If both have permitDate within 3 days of each other AND address similarity > 0.5 (lower threshold than standalone), consider it a match. This catches cases where two portals format addresses slightly differently.

Updated logic:
```typescript
export function isLikelyDuplicate(a: DedupCandidate, b: DedupCandidate): boolean {
  // Cannot dedup without coordinates on both leads
  if (a.lat == null || a.lng == null || b.lat == null || b.lng == null) {
    return false;
  }

  // Step 1: Geographic proximity check (unchanged)
  const distance = haversineDistance(a.lat, a.lng, b.lat, b.lng);
  if (distance > PROXIMITY_THRESHOLD_MILES) {
    return false;
  }

  // Step 2a: Permit number match (cross-source signal)
  if (a.normalizedPermitNumber && b.normalizedPermitNumber) {
    const permitSim = compareTwoStrings(a.normalizedPermitNumber, b.normalizedPermitNumber);
    if (permitSim > 0.8) return true;
  }

  // Step 2b: Original text similarity (address OR title)
  const addressSim = compareTwoStrings(a.normalizedAddress, b.normalizedAddress);
  const titleSim = compareTwoStrings(a.normalizedTitle, b.normalizedTitle);
  if (addressSim > SIMILARITY_THRESHOLD || titleSim > SIMILARITY_THRESHOLD) {
    return true;
  }

  // Step 2c: Date + address compound match (cross-source signal)
  if (a.permitDate && b.permitDate) {
    const daysDiff = Math.abs(a.permitDate.getTime() - b.permitDate.getTime()) / (1000 * 60 * 60 * 24);
    if (daysDiff <= 3 && addressSim > 0.5) {
      return true;
    }
  }

  return false;
}
```
