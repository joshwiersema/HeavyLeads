<objective>
Replace Haversine full-table-scan distance filtering with PostGIS ST_DWithin spatial index queries and add SQL-level LIMIT to getFilteredLeadsWithCount.

Purpose: The lead feed queries currently compute Haversine distance per row in WHERE clauses, ignoring the existing GiST index on leads.location. The location column is never populated. This plan populates it during pipeline insert and rewrites all distance-filtering queries to use ST_DWithin, which leverages the spatial index for O(log n) lookups instead of O(n) per-row computation. Additionally, getFilteredLeadsWithCount fetches ALL within-radius rows and paginates in memory -- this gets a separate COUNT query + SQL LIMIT.

Output: Spatial-index-backed queries across all lead feed and storm alert functions, SQL-level pagination in getFilteredLeadsWithCount.
</objective>

**A. src/lib/leads/queries.ts -- Replace Haversine with ST_DWithin in all 3 functions:**

For all three functions (`getFilteredLeads`, `getFilteredLeadsWithCount`, `getFilteredLeadsCursor`):

1. Replace the Haversine distance SQL in the WHERE clause:

OLD (appears 3 times, once per function):
```sql

```

NEW -- use ST_DWithin with the location column. ST_DWithin on geography type uses meters, so convert miles to meters (* 1609.344):
```typescript

  ${radiusMiles * 1609.344}
)`
```

2. Keep the `isNotNull(leads.lat)` and `isNotNull(leads.lng)` conditions but ADD `isNotNull(leads.location)` so the spatial index can be used. Leads without location populated won't appear until backfill runs.

```typescript
const distanceExpr = sql<number>`
  ST_Distance(

`.mapWith(Number);
```

This returns miles (ST_Distance returns meters for geography, divide by 1609.344).

**B. src/lib/leads/queries.ts -- Add SQL LIMIT to getFilteredLeadsWithCount:**

Currently `getFilteredLeadsWithCount` (lines 480-633) fetches ALL within-radius leads with no LIMIT, enriches them all in memory, then slices. Change to:

1. Run a separate COUNT query first:
```typescript
const [{ count: totalCount }] = await db
  .select({ count: sql<number>`count(*)`.mapWith(Number) })
  .from(leads)
  .where(and(
    isNotNull(leads.location),
    sql`ST_DWithin(...)`,
    ...filterConditions
  ));
```

2. Then run the data query with SQL-level `.limit(pageSize).offset((page - 1) * pageSize)`:
```typescript
const rows = await query
  .where(and(...conditions))
  .orderBy(desc(leads.scrapedAt))
  .limit(pageSize)
  .offset((page - 1) * pageSize);
```

3. Compute `totalPages = Math.ceil(totalCount / pageSize)` from the COUNT result.

4. Still enrich and score the fetched page of rows in memory, but now only `pageSize` rows (typically 20-50) instead of potentially thousands.

NOTE: Because scoring happens in memory, the SQL-level pagination means results are sorted by `scrapedAt DESC` at the SQL level, not by score. This is acceptable because `getFilteredLeadsWithCount` is only used by tests (the dashboard uses `getFilteredLeadsCursor`). If score-based sorting is needed in the future, the scoring would need to move to SQL.
